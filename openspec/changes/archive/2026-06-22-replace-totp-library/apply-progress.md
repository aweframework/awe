## Post-Archive Auth Hardening

- **Issue**: Final fresh reviews found two backend authorization gaps after the original replacement work: `/access/qr-code` did not enforce a fail-closed full-2FA gate, and FORCE-mode users without enrolled 2FA could still be backend-authenticated before enrollment completed. ## Post-Verify Remediation: separate settings preservation from force/login activation

- **Issue**: Treating every authenticated non-2FA secret generation as a settings activation was unsafe. The force/login activation path also reaches `generate2faSecret()` through `getQRCodeList(true)`, so setting `fullyAuthenticated=true` there could bypass required TOTP verification before the code is entered.
- **Fix**: `TotpService` now routes secret generation through a private helper with an explicit `preserveAuthenticatedSession` flag. Public `generate2faSecret()` enables preservation for settings activation only, while `getQRCodeList(true)` calls the helper with preservation disabled for force/login activation. Store-failure no-mutation behavior remains intact.
